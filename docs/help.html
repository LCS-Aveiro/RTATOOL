RTA Tool — Language Reference

The RTALang Handbook

Everything you need to model, simulate, and verify Reconfigurable Timed Automata. From basic states and clocks to dynamic reconfiguration rules and PDL formulae.

Section 01

What is RTA?

RTA (Reconfigurable Timed Automata) lets you describe a system as a set of states connected by transitions, governed by continuous clocks. On top of that, you can add hyper-edges (rules) that dynamically activate or deactivate transitions — making the system reconfigurable at runtime.

⏱

Timed

Declare continuous clocks, state invariants, and transition guards to enforce real-time constraints.

⚡

Reconfigurable

Hyper-edges allow transitions to dynamically turn each other ON or OFF during execution.

🔒

Guarded

Any edge can carry an if (condition) guard evaluated before it is allowed to fire.

📐

Verifiable

Export models to UPPAAL for model checking, or use built-in Propositional Dynamic Logic (PDL).

Section 02

File Structure

An .rta file is plain text. The order of declarations is flexible — but a recommended layout is shown below.

timer-model.rta
// 1. Optional model name
name MyModel

// 2. Clocks and variables
clock t
int   counter = 0

// 3. Initial state and invariants
init start
inv  start : t <= 10

// 4. Transitions
start ---> end : timeout if (t >= 10) then { t' := 0 }

// 5. Hyper-edges (rules)
timeout --! timeout : disableSelf
Section 03

Time, Clocks & Invariants

RTA extends traditional automata with dense time. Time passes automatically, increasing the value of all declared clocks at the same rate, unless constrained by a state invariant.

Clocks

Clocks are continuous variables that start at 0.0.

clock x
clock y, z

// You can reset clocks during transitions:
s0 ---> s1 : resetTime then { x' := 0 }

Invariants

Invariants dictate how long the system is allowed to stay in a given state. If time passing would violate the invariant, the system must take a transition before that happens.

inv working : x <= 5
// The system cannot stay in 'working' if clock x > 5.
Section 04

Transitions & Edges

A transition moves the system from one state to another. The canonical syntax is:

SOURCE ---> TARGET : label [disabled] [if (condition)] [then { updates }]

Arrow styles

ArrowMeaning
--->Standard transition (shorthand, auto-ID = label)
-transId->Transition with an explicit transId distinct from the label
transId vs label When you write s0 -step-> s1 : myLabel, the edge has transId = step and label = myLabel. Using ---> makes both equal. The label is what rules target; the transId is used to uniquely identify the edge in the graph.

Basic examples

s0 ---> s1 : go

// With named transId
s0 -t1-> s1 : myAction

// Starts disabled (inactive until a rule enables it)
s0 ---> s1 : secret disabled

// Self-loop
s0 ---> s0 : idle
Section 05

Hyper-Edges (Rules)

Rules are directed edges between labels, not states. When the source label fires, the rule dynamically changes the topology of the automata by turning the target label ON or OFF.

ArrowNameEffect
->>ActivationEnables the target edge (turns it ON).
--!DeactivationDisables the target edge (turns it OFF).
--xDeactivation (alias)Same as --!
---->Strong Dep.Adds an ON rule to the target, and an OFF rule to itself.
--#--Mutual Exc.Adds cross-disabling OFF rules between both labels.

Syntax

triggerLabel ->> targetLabel [: ruleLabel] [disabled] [if (condition)] [then { updates }]
rules.rta
// Every time 'unlock' fires, it activates the 'open' transition
unlock  ->>  open : rule1

// Every time 'lock' fires, it deactivates 'open'
lock    --!  open : rule2

// Rule only fires if a condition is met
login   ->>  premiumFeature if (userTier >= 2)

// Rule that also updates a variable when triggered
purchase ->> upsell if (cart > 0) then {
    revenue' := revenue + 1
}
Cascade propagation Rules can target other rule labels, creating chains. If a ->> b and b ->> c, firing a will cascade through b and reach c.
Section 06

Variables & Types

Scalar types

int   count    = 0           // integer, default 0
int   level[0..10]    = 1   // bounded integer [min..max]
float ratio    = 0.5
float temperature[0.0..100.0] = 20.0 
bool  isActive = false

Arrays

// Static array
int[] scores = [10, 20, 30]

// Dynamic array (grows when you assign to index == length)
dyn int[] log = []

// Dynamic with max size
dyn int[] recent [max=5] = []   // keeps only last 5 items

// Access in an expression
x' := scores[1]          // x = 20

// Write
scores[0]' := 99         // scores becomes [99, 20, 30]
Section 07

Conditions (Guards)

A guard is an if (...) clause attached to any edge or rule. The transition only fires if the condition is true in the current state.

Comparison operators

OperatorMeaning
==Equal (also accepted as =)
!=Not equal
<Less than
<=Less than or equal
>Greater than
>=Greater than or equal

Logical connectives

KeywordSymbolMeaning
AND&&Both conditions must hold
OR||At least one must hold

Examples

// Clock guard
s0 ---> s1 : timeout if (t >= 10)

// Compound guard (clock + variable)
s0 ---> s2 : jump  if (t >= 5 AND score >= 10)

// Guard with then-block
s0 ---> s1 : advance if (n < 3) then {
    n' := n + 1
}
Section 08

Updates (Side Effects)

When a transition fires, it can update variables and reset clocks. Updates appear in a then { ... } block, or inline for a single assignment.

Assignment

// Inline
s0 ---> s1 : pay  total' := total + 10

// then-block for multiple statements (like resetting a clock)
s0 ---> s1 : buy  then {
    balance' := balance - 5
    t'       := 0
}

// Nested if inside then
s0 ---> s1 : check then {
    if (score > 50) then {
        grade' := 1
    }
}

Math expressions

OperatorMeaning
+Addition
-Subtraction
*Multiplication
/Division

Print statement

print(counter)        // prints value to browser console
Section 09

Functions

You can define reusable functions and call them inside expressions or updates.

def clamp(x, lo, hi) {
    if (x < lo) then { return lo }
    if (x > hi) then { return hi }
    return x
}

s0 ---> s1 : adjust then {
    volume' := clamp(volume + 10, 0, 100)
}

// foreach loop (works on arrays)
def sumArr(arr) {
    total' := 0
    foreach (item in arr) {
        total' := total + item
    }
    return total
}
Section 10

Modules — aut

The aut keyword creates a named sub-automaton. State and label names inside are automatically prefixed, so there are no name collisions between modules.

aut RobotA {
    init idle
    idle    ---> working : start
    working ---> idle    : stop
}

aut RobotB {
    init idle
    idle    ---> busy    : start disabled
    busy    ---> idle    : done
}

// Cross-module rule: RobotA's 'start' enables RobotB's 'start'
RobotA.start ->> RobotB.start : sync
Section 11

PDL Verification

The PDL (Propositional Dynamic Logic) verifier checks logical properties of your model starting from any state.

Propositional connectives

true  |  false  |  ¬φ  |  φ ∧ ψ  |  φ ∨ ψ  |  φ → ψ  |  φ ↔ ψ

State & condition propositions

s0 // true if current state = s0
counter == 3 // variable condition
t <= 5 // clock condition

Modal operators

FormulaMeaning
<> φDiamond: some next state satisfies φ
[] φBox: all next states satisfy φ
<α> φAfter program α there exists a state satisfying φ
[α] φAfter program α all reachable states satisfy φ

PDL programs (α)

SyntaxMeaning
actionNameFire this action once
delay:1.0Explicitly let 1.0 time unit pass
a ; bSequence: fire a then b
a + bChoice: fire a or b
a*Kleene star: repeat a zero or more times
Where to type PDL Use the PDL tab in the right panel. Select a starting state, then type your formula. Click the coloured buttons to insert state names, actions, and operators automatically.

PDL examples

// Can we reach state 'win'?
<> win

// Is it true that every next step goes to 'safe'?
[] safe

// After firing 'timeout', can we reach 'end'?
<timeout> end

// No matter how we fire 'move', we stay in 'valid'?
[move*] valid
// After firing 'a' then 'b', is 'target' reachable?
<a ; b> target

// After letting 5.0 time units pass, is 'timeout' available?
<delay:5.0> <timeout> true

// After any number of 'loop' firings, is 'done' reachable?
<loop*> done
// Does every reachable state keep score >= 0?
[] score >= 0

// Combination: in state 'end' with clock t > 10
end && t > 10
Section 12

LTL Verification

Linear Temporal Logic (LTL) checks properties over a single infinite/finite trace of the model, rather than a branching tree of states. RTA Tool evaluates LTL either over a randomly generated trace (fast, approximate) or exhaustively via a symbolic lasso search (slower, provides real proofs / counter-examples).

Same input box as PDL LTL and PDL formulae share the same input field. The tool auto-detects LTL by the presence of the X, U, G or F operators — if none of these appear, the formula is parsed as PDL instead.

Propositional connectives

true  |  false  |  ¬φ  |  φ ∧ ψ  |  φ ∨ ψ  |  φ → ψ  |  φ ↔ ψ

Edge / hyperedge & condition propositions

Note: unlike PDL, LTL propositions do not refer to the current location/state. A bare name checks whether that edge or hyperedge label is currently active (i.e. present in the model's act set) — clocks and data variables are checked separately via [ ] conditions.

open // true if the 'open' edge/rule label is currently active
[counter == 3] // data variable condition
[t <= 5] // clock condition

Temporal operators

SymbolNameMeaning
X φNextφ holds in the very next step of the trace
G φGloballyφ holds in every step from here onward
F φEventuallyφ holds at some step from here onward
φ U ψUntilφ holds until ψ becomes true (ψ must eventually hold)

Evaluation modes

ModeDescription
Single traceGenerates one random trace of N steps and evaluates φ over it.
BatchGenerates many distinct random traces; reports how many satisfy φ and returns the first counter-example found, if any.
Exhaustive (symbolic)Explores the full symbolic (zone-based) state space up to a limit, searching for a lasso (loop) that violates φ. If the space is small enough, the result is a real proof, not just a sample.

LTL examples

ltl-examples.txt
// Whatever happens, the 'finish' edge always eventually fires again
G F finish

// Once 'lock' fires, 'open' never becomes active again
G (lock -> G !open)

// The counter variable stays non-negative forever
G [counter >= 0]

// 'pending' rule stays active until 'done' fires
pending U done

// In the next step, the 'safe' edge is active
X safe

// Clock guard: t never exceeds 10 along the whole trace
G [t <= 10]
Same input box as PDL LTL and PDL formulae share the same input field. The tool auto-detects LTL by the presence of the X, U, G or F operators. Unlike PDL's StateProp (which can mean "currently in this location"), LTL's bare-name propositions test whether an edge/hyperedge label is active — combine with [ ] conditions to check clocks or variables.
Section 13

CTL Verification

Computation Tree Logic (CTL) reasons about the full branching tree of reachable states rather than a single trace. Every temporal operator is paired with a path quantifier: E (there exists a path) or A (for all paths). Verification is exhaustive and symbolic (zone-based), exploring the reachable state graph up to a state limit.

Propositional connectives

true  |  false  |  ¬φ  |  φ ∧ ψ  |  φ ∨ ψ  |  φ → ψ  |  φ ↔ ψ

Existential operators (E)

FormulaMeaning
EX φSome next state satisfies φ
EF φSome reachable state satisfies φ
EG φThere is a path where φ holds forever
E[φ U ψ]There is a path where φ holds until ψ becomes true

Universal operators (A)

FormulaMeaning
AX φEvery next state satisfies φ
AF φEvery path eventually reaches a state satisfying φ
AG φφ holds in every reachable state, on every path
A[φ U ψ]On every path, φ holds until ψ becomes true

Hybrid binder (↓)

CTL in RTA Tool supports a hybrid down-arrow binder, letting a formula refer back to "the state where we started evaluating".

↓x . φ  // binds x to the current state, then evaluates φ 
// Can we return to the exact state we started from?
↓x . EF x

CTL examples

// Is 'win' reachable from here?
EF win

// No matter what happens, can we always still reach 'win'?
AG EF win
// counter never goes negative, on every path, forever
AG [counter >= 0]

// there's some infinite path that stays in 'idle' forever
EG idle
// On every path, 'loading' holds until 'ready' becomes true
A[loading U ready]

// There is some path where 'retry' holds until 'success'
E[retry U success]
Counter-examples When an AG formula fails, or an EF formula succeeds, the tool reconstructs a shortest witness path via BFS over the explored symbolic state graph and displays it alongside the result.
Section 13

Path Optimizer

The Path tab in the bottom panel uses a breadth-first search to find the shortest path from the initial state to a target.

OptionMeaning
Reach StateFinds the shortest sequence of actions to reach a specific state.
Variable ConditionFinds the shortest sequence of actions until a specific boolean condition holds (e.g. counter == 5).
Section 14

Complete Model Examples

1. Simple Timer

A fundamental example using clocks, invariants, and time-based guards.

timer.rta
name TIMER
clock t
init s0

// Cannot stay in s1 if t > 10
inv s1: t <= 10

s0 ---> s1: start then { t' := 0 }

// Forced to take this before t exceeds 10
s1 ---> s2: timeout if (t >= 10)

// Premature exit
s1 ---> s0: escape if (t < 5)

2. Reconfigurable Content Feed

feed.rta
name LikeAlgorithm
init Feed

Feed    ---> Watch  : watch
Watch   ---> Watch  : like
Watch   ---> Feed   : dontLike
Watch   ---> Feed   : refresh   disabled
Feed    ---> List   : watchLike disabled
List    ---> Watch  : watch2

// Rules
watch     ->>  dontLike  : wd
like      --!  dontLike  : ld
like      ->>  refresh   : lr
like      ->>  watchLike : lw
dontLike  --!  watchLike : dw
